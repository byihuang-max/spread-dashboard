# 三端对齐操作手册

> 适用场景：本地 Mac、GitHub、腾讯云三端 git 状态不一致时的标准修复流程

## 三端角色

| 端 | 角色 | 说明 |
|---|---|---|
| 本地 Mac | 研发机 + pm cron | 晚间全量更新，产出新数据和代码 |
| GitHub | 中心仓库 | 单一事实来源，所有端以它为准 |
| 腾讯云 | 部署机 + am cron | 只做 `git reset --hard origin/main`，不产生本地 commit |

## 分叉常见原因

- 本地 pm cron 跑完 push 时网络不通（GitHub 443 超时）
- 腾讯云 am cron 先 push 了新数据，本地再 push 被 reject
- 本地改了代码但忘了 push，腾讯云又跑了一轮

## 标准对齐流程

### 第一步：开代理
- 本机开 Clash（端口 7897）
- 目的：确保能连 GitHub

### 第二步：本地 fetch + rebase + push
```bash
cd ~/Desktop/gamt-dashboard

# 1. 拉远端最新状态
git fetch origin

# 2. rebase：把本地新 commit 接到远端最新后面
git pull --rebase origin main

# 3. 如果有冲突：解决冲突 → git add → git rebase --continue
# 如果没冲突：直接下一步

# 4. 推到 GitHub
git push origin main
```

> 为什么用 rebase 不用 merge：保持线性历史，避免无意义的 merge commit

### 第三步：关代理
- 关掉 Clash
- 目的：SSH 到腾讯云需要直连（代理开着时 SSH 不通）

### 第四步：腾讯云同步
```bash
ssh ubuntu@111.229.129.146

cd /home/ubuntu/gamt-dashboard
git fetch origin && git reset --hard origin/main

# 验证
git log --oneline -3
```

> 腾讯云始终用 `reset --hard`，不做本地 commit，保持为 GitHub 的纯镜像

### 第五步：验证三端一致
```bash
# 本地
git -C ~/Desktop/gamt-dashboard rev-parse HEAD

# 腾讯云（SSH）
git -C /home/ubuntu/gamt-dashboard rev-parse HEAD

# GitHub（API）
curl -s https://api.github.com/repos/byihuang-max/spread-dashboard/commits/main | grep sha | head -1
```

三个 hash 一致 = 对齐完成。

## 注意事项

- **代理与 SSH 互斥**：开代理时 SSH 到腾讯云不通，关代理才能 SSH
- **腾讯云不要产生本地 commit**：如果服务器上有未提交的改动，先 `git stash` 或 `git checkout .` 再 reset
- **GitHub token 有效期**：当前 token 约 2026-04-26 到期，过期后 push 会报 Authentication failed
- **如果 rebase 冲突太多**：可以 `git rebase --abort`，改用 `git push --force-with-lease`（谨慎，会覆盖远端）

## 什么时候需要 push，什么时候不需要

| 变更类型 | 是否需要 push 到 GitHub | 原因 |
|---|---|---|
| 日常数据更新（CSV/JSON） | 不需要（腾讯云自己跑 cron 就够） | `dashboard.gamtfof.com` 走腾讯云后端，数据由腾讯云本地 `update_all.py` 产出，不依赖 GitHub |
| 代码/页面结构改动（.py / .html） | 需要 | 腾讯云是 GitHub 的镜像，代码改动必须 push → 腾讯云 pull 才能生效 |
| 新增模块/注册表改动 | 需要 | `module_registry.py` 等核心配置改了，腾讯云不 pull 就跑不到新逻辑 |
| GitHub Pages / Cloudflare Pages 展示 | 需要 | 纯静态托管，内容直接来自 GitHub 仓库 |

简单记：**数据自动刷，代码要手推。**

## 自动化建议

理想状态下不需要手动对齐。保持以下纪律即可：
1. 本地 pm cron push 失败时，下次 am 前手动补推
2. 腾讯云 `deploy_sync.sh` 只做 `fetch + reset --hard`，永远不 commit
3. 同一时间只有一端在 push（本地和腾讯云的 cron 时间错开）
