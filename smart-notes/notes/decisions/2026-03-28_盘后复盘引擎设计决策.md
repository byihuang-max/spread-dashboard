# 盘后复盘引擎——盘中涨停因子的解释层

## 一句话
盘后复盘不是一个独立模块，它是**盘中涨停因子**的 explainability 页面：先把今天发生了什么说清楚，再考虑量化。

## 核心设计原则

1. **先描述事实，不先打分** —— 在逻辑没自洽之前，所有打分都是浪费时间
2. **不绑定指数收益** —— 情绪是情绪，指数是指数，不能混在一起
3. **规则引擎，不用黑盒 ML** —— 保证可读、可调、可解释

## 四个事实模块

| 模块 | 回答什么问题 | 数据源 |
|------|------------|--------|
| 板位结构变化 | 今天涨停多了还是少了？首板/二板/三板怎么分布？晋级率？ | `_cache/{date}.json` |
| 高位票跟踪 | 昨天 ≥3 板的高位票今天继续封住了吗？ | `_cache` 交叉比对 |
| 昨日龙头今日表现 | 吐池里的龙头今天怎么走？谁进谁出？ | `leader_pool_history` |
| 核心方向变化 | 今天前五方向和昨天比，延续/分化/切换？ | `_cache` 行业统计 |

## 页首一句话摘要
规则引擎自动生成，示例：
- `涨停大幅扩张，化学制药、电力方向延续。`
- `涨停大幅缩量，高度断崖回落，高位全部断板，方向切换。`

## 文件架构

```
研发主文件（quant-backtest 侧）
├── timing_model/narrative/narrative_engine.py  ← 计算引擎
├── timing_model/narrative/generate_replay_page.py  ← HTML 生成
└── timing_model/export_leader_pool_history.py  ← 吐池历史导出

展示层（gamt-dashboard 侧）
├── timing-research/intraday_limit_replay.html  ← 前端展示页
├── timing-research/index.html  ← 入口按钮（盘中发现层原型卡片）
├── timing-research/generate_replay_page.py  ← 桥接脚本
├── timing-research/export_leader_pool_history.py  ← 桥接脚本
└── server/module_registry.py  ← momentum_stock.post_inject
```

## 自动更新链路
挂在 `momentum_stock` 的 `post_inject`，不需要单独 cron：
1. `generate_leader_pool.py` → 龙头观察池最新页
2. `export_leader_pool_history.py` → 吐池历史导出
3. `generate_replay_page.py` → 盘后复盘 HTML

## 关键教训
- **云端路径不带 `Desktop`**：本地 Mac 是 `~/Desktop/xxx`，腾讯云是 `~/xxx`
- 所有跨机器脚本必须做双路径兼容（`resolve_gamt_root()` 或 `CANDIDATES` 列表）
- `sync-to-cloud.sh` 排除 `*.html/*.json`，产物由云端 cron 自己生成
