# 自选股跟踪器：模块数据源清单（2026-04-12）

## 背景

桌面项目：`~/Desktop/自选股跟踪器`

这条线当前已经明确不再追求“所有模块都硬切到同一种接口”，而是采取更实用的分层方案：

- **行情层**：优先走 `iFind direct API`
- **事件层**：先保留可用方案，不强行全改成 direct API

原因很简单：已经实测确认，`iFind direct API` 对 **行情/量价** 这类接口是通的，但对 `公告 / 新闻检索` 这类事件型入口，当前没有找到和 MCP 等价的现成 HTTP endpoint；直接尝试 `search_notice` / `search_news` 返回 `404`。

## 核心依赖文件

### 1）行情层依赖
- `src/utils/ifind_client.py`

当前已确认可用的 direct API 包括：
- `get_access_token`
- `real_time_quotation`
- `cmd_history_quotation`

已经实测：
- `688262.SH`（国芯科技）的实时行情可拉
- 历史行情与成交量可拉

### 2）事件层依赖
- `src/utils/ifind_mcp_client.py`

当前事件层模块仍通过 MCP 拿：
- 公告片段
- 新闻片段
- 一部分事件型文本结果

但问题也很明确：
- 返回偏文本/表格片段
- 更适合“先拿到结果”
- 不够适合后面做稳定工程化抽取

## 当前技术判断

### 已经清楚的事
1. **行情层适合 direct API**
2. **事件层暂时不能强行全 direct API**
3. 当前项目最合理的路线是：
   - 先把分层口径定住
   - 再逐步优化事件层，而不是继续在“全 direct API”上死磕

### 为什么不继续硬拗“全 direct API”
因为实测中：
- `real_time_quotation`、`cmd_history_quotation` 可用
- 但 `search_notice`、`search_news` 在 direct API 路径下返回 `404`

这说明至少在目前已掌握的 HTTP 接口里：
- 行情接口存在
- 事件检索接口并没有以同样方式暴露出来

所以如果现在强行把事件层都改成 direct API，大概率不是在推进项目，而是在和接口清单死耗。

## 项目当前正式口径

`~/Desktop/自选股跟踪器` 现在按下面的结构推进：

### A. 行情层（保留 direct API）
目标模块：
- 股价变动
- 量价关系

特点：
- 结构清晰
- 更适合定时任务
- 更适合后面飞书直推稳定运行

### B. 事件层（先分开）
目标模块：
- 公告
- 舆情
- 股东行为
- 质押
- 资金异动
- 大单成交

当前策略：
- 先保留可用事件源
- 先把日报结果稳定做出来
- 后续如果找到 direct API 的正式事件接口，再逐步替换

## 对后续开发的实际意义

这份清单的价值不是“记接口名”，而是防止后面继续绕回去。

后面只要记住一句话：

> **行情层已经能 direct API 化；事件层先别死磕统一，先保证可用和稳定。**

这样才能更快推进到下一步：
- 单票调试稳定
- 多票股票池扩展
- 定时任务
- 飞书推送给领导

## 一句话总结

到 2026-04-12 这一步，`自选股跟踪器` 的最优推进方式已经不是“全都改成一种接口”，而是：

> **行情层用 iFind direct API，事件层单独保留可用源，先把系统做成。**
